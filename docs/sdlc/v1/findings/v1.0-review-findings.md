# SDLC v1.0 — Review Findings

**Date**: 2026-04-03
**Reviewer**: Claude (AI pair) — 5-pass structured review
**Document Reviewed**: `docs/sdlc/v1/v1.0-ai-native-sdlc.md`
**Status**: Documented — pending incorporation into living SDLC

---

## Review Methodology

5 sequential passes, each with a different lens:

1. **Structural completeness** — gaps in phase coverage and flow
2. **AI-native authenticity** — is this truly AI-native or traditional SDLC with AI roles?
3. **Practical weight** — is this followable for a solo dev + AI?
4. **Flexibility & evolution** — does it support iteration and adaptation?
5. **Edge cases & anti-patterns** — what scenarios break the process?

---

## Pass 1: Structural Completeness

### 1.1 — No backtracking mechanism [CRITICAL]

**Finding**: Phase dependencies are strictly linear (0→1→2→3→4→5→6→7→1). The only "go back" is Phase 7 → Phase 1.

**Problem**: What happens when you're mid-execution (Phase 5) and realize the spec (Phase 3) is wrong? Or mid-spec writing and discover a new architectural question (Phase 2)? There's no re-entry mechanism.

**Recommendation**: Define explicit re-entry points. E.g., "If execution reveals a spec gap, pause execution, file a finding, return to the minimum necessary phase to resolve, then resume."

### 1.2 — No scope classification [CRITICAL]

**Finding**: "No phase can be skipped" (line 93) applies equally to "build the metrics engine" (weeks of work) and "fix a header alignment bug" (5 minutes).

**Problem**: An 8-phase cycle for a typo fix is process theater. It makes the SDLC feel burdensome and encourages people to skip it entirely.

**Recommendation**: Define scope tracks — Feature (full 8 phases), Enhancement (compressed), Fix (minimal), Hotfix (emergency) — each with defined required phases.

### 1.3 — Phase 3 vs Phase 4 boundary blurry [MEDIUM]

**Finding**: Phase 3 says "spec says *what*, not *how*." Phase 4's task format includes complete implementation code. But the draft Plan 2 has full type definitions, function signatures, and implementations — is that spec or plan material?

**Problem**: Without a clear boundary, plan authors bake spec-level decisions into plans without proper review, or spec authors over-specify implementation details.

**Recommendation**: Clarify: Spec defines interfaces, behaviors, and constraints. Plan defines file paths, task ordering, and TDD steps. Implementation code lives in neither — it's discovered through TDD during execution.

### 1.4 — Spec versioning undefined [MEDIUM]

**Finding**: When a new plan needs details not in the existing spec, there's no guidance on whether to: (a) amend the spec, (b) write a plan-specific supplemental spec, or (c) include it in the plan.

**Problem**: Without guidance, spec coverage drifts — some plans have implicit specs baked in, others reference a stale master spec.

**Recommendation**: Define a spec evolution policy. E.g., "The master spec is the source of truth. Plans may NOT introduce spec-level decisions. If a plan needs something not in the spec, the spec must be updated first (Phase 3 re-entry)."

### 1.5 — Phase 6 missing automated checks [LOW]

**Finding**: Phase 6 (Review + Ship) mentions spec compliance and code quality review but not CI, type checking (`tsc --noEmit`), or lint.

**Recommendation**: Add automated verification as the first step of Phase 6: "Run full test suite, type check, lint, and build before any human/AI review."

## Pass 2: AI-Native Authenticity

### 2.1 — No session/context management [CRITICAL]

**Finding**: This is an *AI-native* SDLC but doesn't address the biggest AI constraint: context windows end. Sessions are lost. Conversations hit token limits.

**Problem**: Without session management, every new conversation starts from scratch. CLAUDE.md helps, but it doesn't capture in-progress state, partial decisions, or execution checkpoints.

**Recommendation**: Add a "Session Management" section covering:
- Checkpoint frequency (e.g., every task completion, every 30 min)
- Minimum viable context to resume (what must be in CLAUDE.md vs what can be re-derived)
- Session handoff protocol (what to save before ending a session)
- Context compression strategy (when artifacts get too large)

### 2.2 — "Complete code in plan" defeats TDD [MEDIUM]

**Finding**: Phase 4 task format includes "Complete implementation code" in the plan. The AI implementer during Phase 5 just copies it.

**Problem**: This isn't TDD — it's transcription. The value of TDD is that the implementation is *discovered* through making tests pass. Pre-writing the code removes that discovery loop. It also means bugs in the plan become bugs in the code without question.

**Recommendation**: Plans should include: (1) test code (the spec for behavior), (2) function signatures and constraints, (3) key algorithmic notes if non-obvious. But NOT complete implementations. Let the AI implementer discover the implementation through TDD.

### 2.3 — No AI discovery feedback loop [MEDIUM]

**Finding**: The SDLC treats AI as a plan-follower during Phase 5. But AI discovers things during execution — edge cases, better approaches, design issues, performance concerns.

**Problem**: If the AI is constrained to "build exactly what the plan specifies, no more, no less" (line 365), these discoveries are suppressed.

**Recommendation**: Add a "Discovery Checkpoint" mechanism. During execution, if the AI discovers something significant: (1) complete the current task, (2) document the discovery, (3) assess impact — cosmetic (note and continue), moderate (flag for post-execution review), critical (pause and escalate to human).

### 2.4 — Multi-agent coordination missing [LOW]

**Finding**: No guidance on when/how to use subagents for parallel execution, or how parallel agents share state during Phase 5.

**Recommendation**: Add a brief section on parallel execution: when to dispatch subagents, how to partition work (by file, by module, by test), how to merge results.

### 2.5 — No model selection guidance [LOW]

**Finding**: Different SDLC phases have different cognitive demands. Architecture decisions need deep reasoning. Boilerplate execution needs speed.

**Recommendation**: Optional — note which phases benefit from which model characteristics (deep reasoning vs fast output), but don't prescribe specific models.

## Pass 3: Practical Weight

### 3.1 — Too heavy for small changes [CRITICAL]

**Finding**: A solo dev + AI running 8 phases for every change will burn out on process overhead.

**Problem**: The SDLC acknowledges this ("the brainstorm can be 5 minutes") but doesn't formalize it. Without formal tracks, the default is always "full process" and people start skipping.

**Recommendation**: Define explicit tracks with required phases per track:

| Track | When | Required Phases |
|-------|------|----------------|
| Feature | New capability, >1 day | All 8 phases |
| Enhancement | Extending existing feature, <1 day | 1 (light), 4, 5, 6 |
| Fix | Bug fix, known root cause | 5, 6 |
| Hotfix | Production emergency | 5 only (retro bundled into next cycle) |
| Refactor | Tech debt, no behavior change | 1 (light), 4, 5, 6 |

### 3.2 — Retro fatigue [MEDIUM]

**Finding**: Phase 7 (Retrospective) after every plan cycle. After 10 plans, that's 10 retros.

**Problem**: Most retros for smooth cycles will say "went fine." The ceremony becomes noise.

**Recommendation**: Lightweight retro by default (3 bullet points: went well, went wrong, change). Deep retro only when: (a) something went significantly wrong, (b) every 3-5 cycles as a batch, or (c) human requests it.

### 3.3 — ADR bloat [MEDIUM]

**Finding**: "Every architectural decision gets an ADR." At current rate (13 ADRs for Plan 1), the project could reach 50+ ADRs quickly.

**Problem**: Navigating 50+ ADRs becomes a burden. The decision log helps but still requires reading individual ADRs for context.

**Recommendation**: Define an ADR threshold: an ADR is warranted when the decision (1) has 2+ viable alternatives, (2) affects multiple files/modules, AND (3) would be non-obvious to a new team member. Below that threshold, a decision log entry suffices.

### 3.4 — "No phase skipped" vs "brainstorm can be 5 minutes" tension [LOW]

**Finding**: The CRITICAL RULE on line 93 says no phase can be skipped. But the text also says phases can be lightweight.

**Recommendation**: Reframe: "Every track defines its required phases. Required phases cannot be skipped but can scale to the change size. A Feature brainstorm might be a 2-hour session; an Enhancement brainstorm might be a 3-sentence confirmation."

## Pass 4: Flexibility & Evolution

### 4.1 — No fast tracks [CRITICAL]

**Finding**: (Overlaps with 3.1) Every change follows the same pipeline. No formal differentiation by scope.

**Recommendation**: See 3.1 track definitions. This is the single most important addition for long-term sustainability.

### 4.2 — SDLC versioning trigger undefined [MEDIUM]

**Finding**: The versioning policy says "process changes result in a new version" but doesn't define what qualifies as a process change.

**Problem**: Without criteria, either everything triggers a version (churn) or nothing does (staleness).

**Recommendation**: Version triggers: (a) new phase added/removed, (b) track definitions changed, (c) core principles changed, (d) 3+ findings from a review incorporated at once. Individual tweaks accumulate in the living doc without a version bump until they reach threshold.

### 4.3 — No plan decomposition guidance [MEDIUM]

**Finding**: When is a plan too big? Plan 2 draft is ~1400 lines. No guidance on splitting.

**Recommendation**: Add a heuristic: "If a plan has >10 tasks or spans >3 independent modules, consider splitting into sub-plans that can be executed and reviewed independently."

### 4.4 — Tech debt has no home [LOW]

**Finding**: The SDLC covers feature development. Refactoring, dependency updates, and tech debt reduction have no defined cycle.

**Recommendation**: Add a "Refactor" track (see 3.1) and note that tech debt items can be queued and batched into dedicated cycles.

### 4.5 — No experiment/spike concept [LOW]

**Finding**: Sometimes you need throwaway code to answer a question mid-cycle. Phase 0 covers initial discovery, but spikes during later phases (e.g., "will this library work?") have no path.

**Recommendation**: Allow "micro-spikes" during any phase: time-boxed (max 30 min), explicitly throwaway, findings documented, no artifacts kept in codebase.

## Pass 5: Edge Cases & Anti-Patterns

### 5.1 — No "pause and re-plan" during execution [CRITICAL]

**Finding**: You're building Task 8 of 14 and realize Tasks 9-14 are based on a wrong assumption.

**Problem**: The SDLC says "Handle blockers — document why and escalate" but doesn't specify the resolution flow. Do you stop? Finish the current task? Re-plan? Go back to brainstorm?

**Recommendation**: Define an execution interrupt protocol:
1. Complete the current task (don't leave half-done work)
2. Commit what's done
3. Document the blocker/discovery
4. Assess: Can the remaining tasks be adjusted in-place (minor)? Or does the plan need re-writing (major)?
5. Minor: adjust remaining tasks, note the deviation. Major: return to Phase 4 (or earlier if needed), re-plan, resume.

### 5.2 — TDD boundary vague [MEDIUM]

**Finding**: "No code without a test first" has a carve-out for "config/setup." But providers have behavior, layout components have rendering logic. Where's the line?

**Recommendation**: Clarify: "Test-first applies to any code with logic, behavior, or user-visible output. Exempt: static configuration files (next.config.ts, tailwind.config.ts, .env), one-line re-exports, and pure type definitions."

### 5.3 — External dependency changes have no path [MEDIUM]

**Finding**: Library updates, API changes, security patches don't fit the feature cycle.

**Recommendation**: Add to the tracks: "Dependency Update" — similar to Fix track but includes: check changelog, run tests, verify build, commit. No brainstorm or spec needed.

### 5.4 — Discovery artifacts bleed into implementation [MEDIUM]

**Finding**: Phase 0 says "prototype scripts saved as reference material." But the draft Plan 2 treats them as implementation source ("ported from proven scripts"), copying logic verbatim.

**Problem**: Reference scripts are untyped, untested, and designed for CLI output. Porting them directly skips the TDD discovery that would surface better designs.

**Recommendation**: Clarify: "Phase 0 artifacts are *inspiration and validation*, not source code. Implementation must be derived through TDD using the spec, not by porting scripts."

### 5.5 — Testing strategy is flat [LOW]

**Finding**: TDD is mentioned throughout but no distinction between unit tests, integration tests, and E2E tests. When does each apply?

**Recommendation**: Brief guidance: "Unit tests for pure functions and computation modules. Integration tests for API routes and data pipelines. E2E tests (Playwright) for critical user flows. Default to unit unless the test requires real I/O or browser interaction."

## Addendum: Findings from Plan 2 Brainstorm (2026-04-05)

### 6.1 — No production feedback → test loop [MEDIUM]

**Finding**: The SDLC covers the build cycle (Phase 0–7) but has no defined path for: user reports bug → reproduce → add regression test → fix → ship. The test suite is treated as a Phase 5 artifact, not a living system that grows from real usage.

**Problem**: Edge cases can't all be predicted upfront. Real users will surface failures that become regression tests. Without a defined feedback loop, bug fixes happen ad-hoc without strengthening the test suite.

**Recommendation**: Define a regression test protocol: (1) user reports issue, (2) reproduce with a failing test, (3) fix to make test pass, (4) test stays permanently. This also connects to the missing "Fix" track (finding 3.1) — bug fixes should follow: reproduce → regression test → fix → review.

**Origin**: Discovered during Plan 2 brainstorm discussion about edge case testing strategy. The principle "ship baseline tests, let real usage tell you what you missed, each bug becomes a regression test" has no home in the current SDLC.
